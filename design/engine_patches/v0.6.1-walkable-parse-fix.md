# Engine bug + fix: v0.6.0 walkable column parsing (→ v0.6.1)

**Found:** 2026-06-07, while wiring paths-only guest movement into the zoo.
**Fixed in engine:** commit `4040ef6`, tagged **v0.6.1**.
**Patch (durable copy):** [`v0.6.1-walkable-parse-fix.patch`](./v0.6.1-walkable-parse-fix.patch)

## The bug

`ContentDB._compile_entities` parsed the v0.6.0 walkable-network columns
(`walkable`, `traversal_cost`, `network_id`, `walkable_tags`) **inside** the
`if row.has("useful_life_days"):` block:

```gdscript
if row.has("useful_life_days"):
    ed.useful_life_days = _cell_int(...)
    if row.has("walkable"):                 # ← only runs when the row ALSO
        ed.walkable = _cell_bool(...)       #   has a useful_life_days column
    ...
```

`useful_life_days` is an optional v0.5.0 accounting column. The zoo's
`entities.md` doesn't use it, so the branch never executed and **no tile was
ever registered as walkable** — every `WalkableNetwork` stayed empty and no
agent could path. The whole v0.6.0 navigation surface was dead for any game
without a `useful_life_days` column.

## The fix (v0.6.1)

Pure indentation: dedent the four `if row.has(...)` walkable blocks one level
so they run for **every** entity row, independent of the accounting column.
No behavior change for tables that already carried `useful_life_days`.

Validated:
- Engine GUT suite: **295/295** green (unchanged).
- Zoo: `test_path_tiles_build_a_walkable_network` now passes — placing `path`
  tiles builds a routable network.

## Why not a zoo-side workaround

A workaround existed (declare a no-op `useful_life_days` column in the zoo's
`entities.md` to un-gate the parse), but it leans on the bug and every other
game on the engine hits the same wall. Per `CLAUDE.md` §1 this is engine work;
fixing it at the source and bumping the tag is the right call (decided with
the owner).

## How to land it (push step this session couldn't do)

From a checkout with write access to the engine remote:

```sh
cd engine
git push origin main         # pushes commit 4040ef6
git push origin v0.6.1       # pushes the tag
```

If the local engine commit was lost (e.g. container reset), reapply the patch
onto `main` first:

```sh
cd engine
git fetch origin && git checkout main
git am ../design/engine_patches/v0.6.1-walkable-parse-fix.patch
git tag -a v0.6.1 -m "v0.6.1 — fix walkable column parsing"
git push origin main v0.6.1
```

The zoo is already pinned to this commit (`git submodule status` →
`4040ef6 (v0.6.1)`). Once it's on the remote, the pin is reproducible and CI
(`.github/workflows/deploy.yml`, `submodules: recursive`) will resolve it.
